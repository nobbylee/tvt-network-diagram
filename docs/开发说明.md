## 1. 项目背景

### 1.1 公司背景

**TVT（同为数码科技，股票代码 002835）** 是一家视频安全领域的产品与系统方案提供商，产品涵盖：

- 网络摄像机（IPC）、AI 摄像机、球机、热成像等
- NVR / DVR / 混合录像机
- 门禁、可视对讲
- NVMS 管理平台及配套服务器
- 解码器、电视墙等显示设备

产品广泛应用于平安城市、交通、金融、教育、园区等行业，业务覆盖全球 120 多个国家。

### 1.2 问题与动机

在客户项目实施过程中，技术支持与研发人员经常需要在本地复现客户的网络环境以排查问题。现有通用绘图工具（Visio、draw.io 等）存在以下不足：

- 缺乏 TVT 及安防行业专用设备图标
- 操作复杂，学习成本高
- 无法结构化记录 IP、端口、连接类型等网络信息
- 不便团队内共享与复用客户拓扑

因此需要开发一款 **面向 TVT 安防产品的专用网络架构图绘制工具**，操作体验接近思维导图，同时支持结构化数据记录，为后续校验与仿真奠定基础。

### 1.3 产品定位

| 项目 | 说明 |
|------|------|
| 产品名称（暂定） | TVT 网络架构图绘制工具 |
| 目标用户 | 技术支持、研发工程师、售前工程师 |
| 核心场景 | 还原客户现场网络拓扑，团队内共享，本地问题复现 |
| 使用环境 | 主要在办公室（内网 Web），偶尔出差（离线文件） |

## 2. 开发策略：分阶段迭代

采用 **A → C → B** 的渐进式路线，优先交付可用价值，逐步加深能力。

| 阶段 | 版本 | 目标 | 核心功能 |
|------|------|------|----------|
| **A** | v1.0 | 专用拓扑绘图 | 画布、元件库、属性编辑、内网存储、.narch 导入导出 |
| **C** | v1.5 | 配置校验 | IP 冲突检测、PoE 功率估算、NVR 通道数校验等 |
| **B** | v2.0 | 网络仿真 | 根据拓扑生成本地 Docker/VLAN 环境 |
| **后续** | v1.x | 文档导出 | PDF 导出（含客户名、日期、图例） |

**原则：** 第一版专注「画好、存好、分享好」，不做校验和仿真。

## 3. 技术方案

### 3.1 方案选型

采用 **方案一：基于成熟图编辑引擎**，具体技术栈如下：

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 前端框架 | React + TypeScript | 组件化开发，类型安全 |
| 画布引擎 | React Flow | 节点-连线拓扑，生态成熟，扩展性好 |
| 状态管理 | Zustand | 轻量，适合画布状态 |
| UI 样式 | Tailwind CSS + shadcn/ui | 现代简洁界面 |
| 后端框架 | Node.js + Fastify | REST API，高性能 |
| 数据库 | PostgreSQL | 项目元数据、用户、分类 |
| 文件存储 | 本地磁盘 / MinIO | .narch 项目文件 |
| 部署 | 内网 Docker | 部署在公司现有内网服务器 |

### 3.2 系统架构

```
┌─────────────────────────────────────────────────────────┐
│                   浏览器（Chrome / Edge）                │
│  ┌──────────┐  ┌──────────────┐  ┌──────────────────┐ │

│  │ 面板     │  │  画布引擎    │  │                  │ │
│  └──────────┘  └──────────────┘  └──────────────────┘ │

                         │ HTTPS
┌────────────────────────▼────────────────────────────────┐
│                    内网服务器（Docker）                   │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
```

### 3.3 选型理由

1. **贴合场景** — 安防拓扑本质是「设备节点 + 连线 + 属性」，React Flow 天然适配
2. **开发效率** — 画布引擎成熟，第一版预计 2–3 个月可交付
3. **扩展性** — 节点携带结构化 JSON 数据，为 C 阶段校验、B 阶段仿真预留接口
4. **体验可控** — 通过定制交互（拖拽、吸附、属性面板）实现类思维导图的简洁操作

### 4.3 元件库

#### TVT 设备（按形态分类，型号手动填写）

| 分类 | 图标变体 | 说明 |
|------|----------|------|
| IPC | 枪机、半球、筒机、海螺、全景 | 按外形区分，不按型号 |
| 球机 | 高速球 / PTZ | — |
| 录像机 | NVR、DVR | — |
| 门禁 | 门禁终端 | — |
| 平台 | NVMS 平台 | 软件平台节点 |
| 服务器 | NVMS 转发 / 存储 / 管理 | 同一图标 + 角色标签区分 |
| 显示 | 电视墙、解码器 | — |

> **设计原则：** 每种设备形态对应一个通用图标，具体型号由用户在属性面板手动输入（如 `TD-9544E3B-A`），不为每个 SKU 单独制作图标。

#### 第三方设备

统一归类为「第三方设备」，不再按海康 / 大华 / 宇视拆分。内置常见形态（枪机、半球、球机、NVR、DVR），用户可在元件库中自行增删改。

#### 元件库自定义

用户可在元件库中：
- **新增**：选择分类（TVT / 第三方 / 通用网络）、名称、图标形态
- **编辑**：修改名称、分类、图标
- **删除**：从元件库移除（不影响画布上已放置的设备）

自定义数据保存在浏览器 localStorage，刷新后保留。

#### 通用网络设备

| 设备 | 说明 |

| 交换机 | 含 PoE 交换机 |
| 路由器 | — |
| 防火墙 | — |
| PC / 工作站 | — |

## 5. 数据模型

### 5.1 .narch 项目文件格式

`.narch` 是 JSON 格式的项目文件，可用文本编辑器打开，便于版本管理与调试。

```json
{
  "version": "1.0",
  "meta": {
    "projectName": "XX银行监控项目",
    "customer": "XX银行",
    "author": "张三",
    "createdAt": "2026-07-08T10:00:00Z",
    "updatedAt": "2026-07-08T14:30:00Z"
  },
  "nodes": [
    {
      "id": "node-1",
      "type": "ipc-bullet",
      "brand": "TVT",
      "model": "TD-9544E3B-A",
      "name": "大门入口摄像机",
      "ip": "192.168.1.101",
      "notes": "",
      "position": { "x": 100, "y": 200 }
    },
    {
      "id": "node-2",
      "type": "nvr",
      "brand": "TVT",
      "model": "TD-3316H2-16P-B2-B",
      "name": "16路PoE NVR",
      "ip": "192.168.1.10",
      "notes": "",
      "position": { "x": 400, "y": 200 }
    }
  ],
  "edges": [
    {
      "id": "edge-1",
      "source": "node-1",
      "target": "node-2",
      "sourcePort": "LAN",
      "targetPort": "Port 1",
      "connectionType": "PoE",
      "notes": ""
    }
  ]
}
```

### 5.2 数据库表结构（SQLite · v1.0）

> v1.0 采用 **SQLite（better-sqlite3）** 便于单机与 Docker 部署，字段语义与原 PostgreSQL 设计一致。拓扑 JSON 直接存于 `projects.diagram` 列。

```sql
CREATE TABLE users (
  id           INTEGER PRIMARY KEY AUTOINCREMENT,
  username     TEXT UNIQUE NOT NULL,
  password     TEXT NOT NULL,
  display_name TEXT,
  created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE projects (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  name       TEXT NOT NULL,
  customer   TEXT,
  author_id  INTEGER REFERENCES users(id),
  diagram    TEXT NOT NULL DEFAULT '{"nodes":[],"edges":[]}',
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 6. API 设计（v1.0）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/auth/login` | POST | 用户登录，返回 JWT Token |
| `/api/auth/me` | GET | 获取当前用户信息 |

| `/api/projects/:id/export` | GET | 下载 .narch 文件 |
| `/api/projects/import` | POST | 上传 .narch 文件导入项目 |

**认证方式：** 第一版采用简单账号密码 + JWT；后续可对接公司 LDAP/AD。

## 10. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| React Flow 性能（节点过多） | 大项目卡顿 | 虚拟化渲染、节点数量软限制提示 |
| 设备图标设计耗时 | 延迟交付 | 第一版用 SVG 简图标，后续迭代美化 |
| 内网部署环境差异 | 部署困难 | Docker 标准化，提供部署文档 |
| 出差离线数据同步冲突 | 数据覆盖 | 导入时提示「覆盖/另存为」，不做自动合并 |
| 用户习惯迁移 | 推广阻力 | 提供 2–3 个典型项目模板，降低上手门槛 |

---

## 11. 成功标准

v1.0 交付时，应满足：

1. 技术支持人员能在 **10 分钟内** 完成一个典型客户拓扑的绘制
2. 项目可保存到内网服务器，团队成员可搜索并打开
3. 项目可导出为 .narch 文件，在离线环境导入并继续编辑
4. 元件库覆盖 TVT 全形态 + 第三方设备 + 通用网络设备，并支持自定义增删改
5. 设备节点可记录名称、品牌、型号、IP、备注；连线可记录端口和连接类型
